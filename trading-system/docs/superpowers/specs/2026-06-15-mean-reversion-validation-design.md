# 설계: 평균회귀(RSI+볼린저) 정직한 OOS 검증 — 전략기반 출구 포함

> 작성일: 2026-06-15 · 대상 크레이트: `crates/strategy`, `crates/api` · 엔진/라이브 런타임/HTTP API 불변
> 사전등록 문서 — **숫자를 보기 전에** 출구 모델·그리드·판정 기준을 확정한다(적대적 설계리뷰 반영판).

## 0. 배경 — 왜 이 작업인가

롤링 돌파 계열(순수 변동성돌파, 추세필터 변동성돌파)은 워크포워드 + OOS + 수수료로
**두 번** 엣지 없음이 실증됐다(순수 OOS +0.00%, 추세필터 OOS 평균 −136.74). 사전등록 STOP에
따라 같은 데이터 위 같은 계열 4번째 노브는 금지. 다음은 **구조적으로 다른 베팅** =
평균회귀(역추세). 기존 `TechnicalStrategy`(RSI+볼린저)는 라이브에 쓰인 적은 없으나
한 번도 OOS 검증을 안 한 후보다.

## 0.1 적대적 설계리뷰 결과 (이 문서가 존재하는 이유)

구현 전 3-렌즈 적대적 리뷰(방법론/메커니즘/스코프)에서 **치명 결함**이 만장일치로 확인됨:

> **백테스트에 전략기반 출구가 없다.** 모든 포지션은 `BasicRiskGate`의 고정 −1%손절/+2%익절
> (2:1 추세지속 브래킷, `crates/risk/src/lib.rs:22-23` 소스 검증)으로만 청산된다. `TechnicalStrategy`는
> 진입 신호만 내고 출구를 정의하지 않는다(`exit_price()`는 SL/TP 터치에만 발화, 시간스톱 없음 —
> `backtest_runner.rs:378-394` 검증).

평균회귀 롱은 급락 직후(RSI≤30 AND close≤하단밴드) 진입하는데, 하니스는 +2% 상승을 요구하며
−1% 손절은 낙하 방향에 그대로 놓인다 = **역추세 진입에 추세지속 브래킷**을 씌운 자기모순.
평균회귀의 진짜 엣지(중간밴드로의 스냅백, 1m에서 보통 1% 미만)는 하니스가 물리적으로 못 잡는다.
→ 결과가 음수여도 "평균회귀 엣지 없음"이 아니라 "잘못된 출구로 테스트함"이 된다.

**따라서 출구 모델을 평균회귀에 맞게 바꾸기 전에는 어떤 OOS 숫자도 무의미하다.**

## 1. 목표 / 성공 기준 (사전등록, goal-driven)

검증 가능 목표 = **평균회귀가 수수료 반영 후 OOS에서 일관 양(+)의 per-trade 기대값**을 갖는가.

### 3진 판정 (숫자 보기 전 확정 — 둘 다 정직)
윈도우당 **최소 closed-trade floor = 20**(미만이면 그 윈도우는 데이터 부족).
- **PASS**: 4개 윈도우 전부 floor 충족 AND 수수료후 per-trade 기대값 > 0 AND OOS 평균 PnL > 2×왕복비용 AND 2×비용에서도 생존.
- **FAIL (no-edge)**: 거래수 충분한데 기대값 ≤ 0 → 깨끗한 falsification → STOP.
- **INCONCLUSIVE**: 한 윈도우라도 floor 미달 → 하니스가 가설을 검증 못 함(엣지 없음 아님). 출구/데이터 문제로 보고.

near-zero라도 **거래수가 충분할 때만** no-edge로 읽는다. floor 미달이면 INCONCLUSIVE.

## 2. 출구 모델 (사전등록 — 핵심 수정)

평균회귀 후보 전용 출구를 **테스트 하니스에** 추가한다(프로덕션 `run_backtest` 출구 불변):
1. **평균 복귀 출구**(주): 롱은 close가 중간밴드(20-SMA) 이상으로 복귀하면 청산, 숏은 이하로 복귀하면 청산. (= "평균으로 돌아왔다, 목표 달성".)
2. **하드스톱**(보호): 진입가 대비 고정 −X%(기본 1%, 비대칭 아님 — 평균회귀는 더 떨어지면 더 매력적이나 무한손실 방지용 보호선). LONG=진입가×(1−stop), SHORT=진입가×(1+stop).
3. **시간스톱**: 진입 후 N봉(기본 60봉=1m 1시간) 안에 1·2 둘 다 안 터지면 강제 청산 → 포지션이 윈도우 끝까지 묶여 재진입을 굶기는 것 방지.

우선순위: 같은 봉에서 하드스톱이 먼저 평가(보수적), 그다음 평균복귀, 그다음 시간스톱.

### 대조군 (사전등록)
같은 진입 + **대칭 1:1 브래킷**(±1% TP/SL) 도 한 번 돌린다 → 음수가 출구 비대칭 탓이 아니라
진입신호 자체 문제임을 분리 확인. 결과는 보조 진단으로만 보고(주 판정은 평균복귀 출구).

## 3. 파라미터 그리드 (사전등록 — 숫자 보고 넓히기 금지)

`rsi_period {7, 14, 21}` × `bollinger_period {14, 20, 30}` × `(oversold, overbought) {(30,70),(20,80),(25,75)}`
= **27 조합**. 볼린저 std 배수 2.0 **고정**(5번째 노브 금지 — 판정은 이 값 조건부임을 명시).
출구 파라미터(중간밴드/RSI50/하드스톱1%/시간스톱60봉)도 **고정**(그리드 대상 아님 — 진입신호만 검증).

## 4. 워크포워드 (직전 하니스와 동일 구조)

4개 롤링 윈도우, IS 4개월 → OOS 2개월, BTC+ETH binance 1m, DB 3년치 적재됨.
- W1: IS 2024-06..10 → OOS 2024-10..12
- W2: IS 2025-01..05 → OOS 2025-05..07
- W3: IS 2025-08..12 → OOS 2025-12..2026-02
- W4: IS 2026-01..05 → OOS 2026-05..현재(clamp)

IS에서 고름(아래 5), OOS로만 보고. 전체 IS 그리드도 출력(plateau vs spike 구분).

## 5. IS 선택자 (사전등록)

raw PnL argmax는 저빈도 전략에서 노이즈에 약함. → **IS 최소거래수(≥20) 충족 조합 중에서**
raw PnL 최대를 고른다(거래 0건 조합이 운으로 IS-best 되는 것 차단). 보조로 expectancy-best도 출력.

## 6. 진단 출력 (윈도우별, 필수)

trades · signals_seen · wins/losses · 수수료후 per-trade expectancy · IS-best PnL 옆 OOS PnL ·
open-at-end count. (직전 하니스는 realized_pnl만 남겨 해석 불가였음 — 전체 metrics 반환.)

## 7. 다중비교 (판정문에 명시)

같은 candle DB·같은 BTC+ETH·같은 4 윈도우(2024~2026 단일 거시국면)에서 **3번째** 전략 패밀리.
4/4 양수는 단일패밀리 ~p=0.06, 3회 시도 후엔 더 약함. marginal PASS면 **한 번도 안 쓴 새 홀드아웃
윈도우**(또는 추세장 vs 횡보장 국면별 분해)로 재검증 후에야 testnet 고려. 평균회귀는 본질상 국면의존적.

## 8. 코드 변경 범위 (외과적 — 라이브 불변)

- **`crates/strategy/src/lib.rs`**: `TechnicalStrategy::new(...)` + getter 추가. `Default` 불변. TDD 신규 테스트(비기본 임계값이 신호를 실제로 바꾸는지). 중간밴드(20-SMA)/RSI 헬퍼는 기존 `calculate_bollinger_bands`/`calculate_rsi`/`simple_moving_average` 재사용.

  손실을 가장 덜 본 것일 뿐, 엣지가 아님.

**판정: FAIL (no edge) → STOP. 그리드 확장·재튜닝 금지.** 라이브 교체 안 함.

OOS 엣지 없음 확정(순수돌파 +0.00%, 추세필터 −136.74, 평균회귀 −185.40).
