코인 선물 AI 하이브리드 트레이딩 시스템 최종 구현계획서

Rust 기반 초고속 실행 엔진, Claude 거시 전략 필터, Candle 로컬 패턴 필터, PostgreSQL 보존 레이어, Next.js 관제 대시보드를 결합한 멀티 거래소 선물 자동매매 시스템 구현 명세입니다.

Version 1.1 작성일 2026-06-12 범위 Binance USDT-M · Bybit · Bitget 출시 정책 Paper-first

0. 실행 요약

목표

거래소별 웹소켓 데이터 수집, AI 필터링, 리스크 제한, OCO 주문 예약, 실시간 관제를 하나의 운영 가능한 시스템으로 구축합니다.

핵심 원칙

실거래보다 검증을 먼저 수행합니다. 모든 주문 경로는 기본적으로 paper 모드에서 동작하고, 명시적 승인 후에만 live 모드로 전환됩니다.

거래소 범위

v1은 Binance USDT-M, Bybit USDT Perpetual, Bitget USDT Futures를 지원하며 공통 어댑터 인터페이스로 분리합니다.

구현 기준: 본 문서는 신규 Rust workspace를 생성해 바로 구현할 수 있도록 모듈, DB, API, 테스트, 운영 게이트를 결정한 최종 작업계획서입니다. 실제 API 키, 계좌 정보, 서버 접속 정보는 포함하지 않습니다.

목차

아키텍처 Rust 구조 거래소 어댑터 PostgreSQL 주문·리스크 AI 필터 대시보드 로드맵 테스트 운영·보안

1. 전체 아키텍처

시스템은 비용이 큰 AI 판단을 상위 필터로 제한하고, 빠른 시장 데이터 처리와 주문 실행은 Rust 이벤트 파이프라인이 담당합니다.

Layer 1 · Macro Brain

Claude API가 거시 뉴스, 주요 지표, 시장 감성을 점수화합니다.

GATE 점수 범위 -100..100, 극단 악재 시 신규 진입 차단.

Layer 2 · Local AI Filter

Candle 기반 로컬 패턴 매칭이 15m, 1h 차트와 과거 유사 구간을 비교합니다.

필수 매칭 승률 65% 이하이면 State::Locked.

Layer 3 · Execution Hand

Tokio, Axum, SQLx 기반 Rust 엔진이 가격 수집, 신호 평가, 주문 예약을 수행합니다.

주의 실거래는 TRADING_MODE=live와 승인 토큰이 모두 필요.

3. 멀티 거래소 어댑터 설계

거래소별 REST, WebSocket, OCO 지원 방식이 다르므로 공통 trait를 먼저 고정하고, 구현체가 거래소별 차이를 내부에서 흡수합니다.

#[async_trait]
pub trait ExchangeAdapter: Send + Sync {
    fn exchange_id(&self) -> ExchangeId;
    async fn subscribe_market_stream(&self, symbols: &[Symbol]) -> Result<MarketStream>;
    async fn fetch_account_snapshot(&self) -> Result<AccountSnapshot>;
    async fn place_market_order(&self, request: MarketOrderRequest) -> Result<OrderAck>;
    async fn place_protection_orders(&self, request: ProtectionOrderRequest) -> Result<ProtectionAck>;
    async fn cancel_order(&self, order_id: ExchangeOrderId) -> Result<CancelAck>;
}
거래소 v1 지원 구현 주의점
Binance USDT-M BTCUSDT, ETHUSDT 우선 서명 쿼리, listen key 갱신, 주문 레이트 리밋, futures endpoint 분리
Bybit USDT Perpetual category 파라미터, TP/SL 설정 API 차이, WebSocket topic 표준화
Bitget USDT Futures productType, marginCoin, plan order 계열 API 검증, 주문 ID 매핑
필수 정책: v1에서는 거래소 간 동시 중복 진입을 금지합니다. 동일 심볼에 대해 하나의 거래소에서 포지션이 열려 있으면 다른 거래소의 신규 진입 신호는 무시하고 risk_events에 기록합니다.

4. PostgreSQL 데이터 모델

실시간 처리에는 메모리 이벤트 버스를 사용하고, 재현성·감사·백테스트를 위해 모든 핵심 판단과 주문 결과를 PostgreSQL에 저장합니다.

테이블 주요 컬럼 용도
candles exchange, symbol, timeframe, open_time, open, high, low, close, volume 전략 계산, 백테스트, 패턴 임베딩 생성
order_books exchange, symbol, event_time, best_bid, best_ask, bid_size, ask_size 슬리피지 추정, 체결 환경 분석
signals id, symbol, side, strategy, score, reason, created_at 전략이 만든 진입 후보 기록
orders id, exchange_order_id, mode, side, order_type, status, price, quantity Paper/Live 주문 상태 추적
positions id, exchange, symbol, side, entry_price, mark_price, leverage, unrealized_pnl 포지션 현황, 청산 조건, 대시보드 표시
risk_events id, severity, rule, action, details, created_at 진입 차단, 킬스위치, 일 손실 한도 도달 기록
ai_decisions id, source, score, decision, model, input_hash, created_at Claude/Candle 판단 감사 로그
backtest_runs id, strategy_version, period_start, period_end, metrics, created_at 출시 게이트 검증 결과 저장

금액·수량 필드는 부동소수점 대신 rust_decimal 또는 정수 tick/lot 단위로 저장합니다.

5. 주문 실행 및 리스크 정책

기본 위험 한도

  • 레버리지: 기본 3x, 최대 5x.
  • 1회 진입 비중: 총 예수금의 최대 5%, 운영 승인 시 최대 10%.
  • 손절: 진입가 기준 -1.0%.
  • 익절: 진입가 기준 +2.0%.
  • 손익비: 최소 1:2 미만 신호는 폐기.

진입 게이트

  • 필수 WebSocket 데이터 지연 2초 이하.
  • 필수 Claude macro score가 금지 구간이 아님.
  • 필수 Candle pattern confidence 65% 초과.
  • 필수 일 손실 한도와 거래소별 포지션 한도 미초과.

주문 흐름

  1. strategy가 신호를 생성하고 risk가 포지션 크기를 산출합니다.
  2. execution이 현재 모드가 paper인지 live인지 확인합니다.
  3. 시장가 진입 주문을 제출하거나, paper 모드에서는 체결 시뮬레이션을 수행합니다.
  4. 체결가 확정 즉시 손절과 익절 보호 주문을 예약합니다.
  5. 보호 주문 등록 실패 시 즉시 포지션 정리 또는 신규 진입 잠금을 수행합니다.
모드 동작 허용 조건
paper 실제 거래소 주문 없이 호가·체결가 기반 가상 주문 생성 기본값. 개발, 백테스트, 모의매매에서 사용
live 거래소 REST API로 실제 주문 제출 TRADING_MODE=live, LIVE_TRADING_APPROVED=true, 운영자 승인 토큰 필요
locked 신규 진입 차단, 기존 포지션 관리만 허용 AI 금지 신호, 킬스위치, 일 손실 한도, 데이터 지연 발생 시 자동 전환

6. AI 레이어 구현

Claude Macro Filter

하루 10~20회 이하 호출을 목표로 거시 리스크만 판단합니다. 입력은 경제 캘린더, 주요 크립토 뉴스, BTC/ETH 주요 변동성 요약으로 제한합니다.

  • 출력: macro_score, risk_level, long_bias, short_bias, halt_reason.
  • 점수 -70 이하 또는 구조적 악재 판단 시 State::Locked.
  • 모든 응답은 ai_decisions에 저장하고 입력 원문은 hash로 추적합니다.

Candle Pattern Filter

과거 3~5년치 15m/1h 캔들 구간을 임베딩하고 현재 패턴과 유사 구간을 검색합니다.

  • 출력: pattern_confidence, historical_win_rate, similar_cases.
  • 신뢰도 65% 이하이면 진입 차단.
  • v1은 BTCUSDT/ETHUSDT부터 학습 데이터 품질을 검증합니다.
비용 통제: 실시간 틱마다 외부 AI를 호출하지 않습니다. Claude는 이벤트 기반 상위 필터로만 사용하고, 실시간 매매 판단은 Rust 로컬 계산과 Candle 점수로 처리합니다.

8. 단계별 구현 로드맵

Phase 1

프로젝트 기반 구축

Rust workspace, 공통 타입, 설정 로더, 로깅, PostgreSQL migration, 기본 Axum health API를 구현합니다.

완료 기준 로컬에서 API 서버 기동, DB 연결, migration 실행, structured logging 확인.

Phase 2

시장 데이터 파이프라인

Binance, Bybit, Bitget WebSocket 연결과 표준 MarketEvent 변환, candles/order_books 저장을 구현합니다.

완료 기준 3개 거래소에서 BTCUSDT 실시간 데이터 수집 및 지연/재연결 테스트 통과.

Phase 3

전략·리스크·Paper 실행

기술 지표 신호, 리스크 게이트, paper 주문 체결 시뮬레이션, 가상 TP/SL 추적을 구현합니다.

게이트 모든 주문이 기본적으로 paper 모드에 기록되고 live API 호출이 발생하지 않아야 합니다.

Phase 4

AI 필터 결합

Claude macro score와 Candle pattern confidence를 전략 진입 게이트에 연결합니다.

완료 기준 AI 차단 사유가 signals, risk_events, ai_decisions에 재현 가능하게 저장됩니다.

Phase 5

대시보드·운영 기능

Next.js 대시보드, 실시간 WebSocket, 포지션/PNL/AI 위험도 화면, 긴급 정지 버튼을 구현합니다.

완료 기준 운영자가 현재 모드, 위험 상태, 포지션 보호 주문 상태를 한 화면에서 확인할 수 있습니다.

Phase 6

Live 전환 준비

소액 실거래 전환 전 백테스트, 모의매매, 장애 주입, API 권한 검증, 킬스위치 검증을 완료합니다.

주의 이 단계 전에는 실거래 API 키를 쓰지 않습니다.

9. 테스트 및 승인 기준

단위 테스트

지표 계산, 포지션 크기 계산, 손절/익절 가격 산출, 거래소 응답 파싱, 에러 매핑을 검증합니다.

통합 테스트

WebSocket 이벤트 → 신호 → 리스크 게이트 → paper 주문 → DB 저장 흐름을 검증합니다.

백테스트

최소 3년치 BTCUSDT/ETHUSDT 구간에서 수익률, MDD, 승률, 손익비, 거래 빈도를 기록합니다.

장애 주입

네트워크 단절, 거래소 429, DB 지연, 보호 주문 실패, AI API 실패를 강제로 발생시켜 locked 전환을 확인합니다.

Paper Trading

최소 2주 이상 실시간 모의매매를 수행하고 주문 누락, 비정상 포지션, 보호 주문 실패가 없어야 합니다.

Live 승인

소액 계정, 출금권한 없는 API 키, 최대 레버리지 제한, 관리자 승인 토큰, 킬스위치 검증 후 제한적으로 허용합니다.

Live 전환 체크리스트

항목 기준 상태
백테스트 전략별 결과가 backtest_runs에 저장되고 MDD 기준 통과 GATE
모의매매 2주 이상 paper 모드 안정 운영 GATE
API 키 출금 권한 없음, IP 제한, 선물 거래 권한만 허용 필수
킬스위치 대시보드와 서버 내부 이벤트 양쪽에서 즉시 locked 전환 가능 필수
레이트 리밋 거래소별 제한을 governor로 통제하고 429 복구 확인 필수

10. 운영·보안·장애 대응

보안

  • API 키는 .env 또는 secret manager로만 주입하고 Git에 저장하지 않습니다.
  • 거래소 API 키는 출금 권한을 비활성화하고 IP allowlist를 적용합니다.
  • 대시보드 제어 API는 관리자 인증과 CSRF 방어를 적용합니다.

관측성

  • tracing으로 주문, 리스크 차단, AI 판단, 재연결 이벤트를 구조화 로그로 남깁니다.
  • 핵심 메트릭: WS 지연, 주문 지연, API 실패율, PNL, locked 상태 지속 시간.
  • 장애 발생 시 risk_events에 운영자가 이해 가능한 사유를 저장합니다.

Fail-safe 기본값

  • DB 쓰기 실패가 지속되면 신규 진입을 차단합니다.
  • 보호 주문 등록 실패 시 즉시 포지션 정리 또는 수동 개입 알림을 발생시킵니다.
  • WebSocket 재연결은 지수 백오프를 사용하되, 장시간 단절 시 locked 상태로 전환합니다.
  • Claude API 실패는 진입 허용이 아니라 보수적 차단으로 처리합니다.
